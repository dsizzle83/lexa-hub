# IEEE 2030.5-2018 digest (standards-coverage audit, LEXA hub)

## Document identity

- **IEEE Std 2030.5-2018** — Smart Energy Profile Application Protocol (SEP 2.0), revision of 2030.5-2013. Schema version "S1" (2013 = "S0", NOT backward compatible, §5.7.2). XML namespace `urn:ieee:std:2030.5:ns`. Normative WADL + XSD in supplemental material; body clauses: 4 design pattern, 5 application support, 6 security, 7 discovery, 8 support resources, 9 common resources, 10 smart-energy resources, 11 proprietary extensions. Annex B = model reference (informative extracts), Annex C examples, Annex D guidelines, Annex E mapping to IEEE 1547-2018.
- Architecture: RESTful HTTP/1.1 (GET/HEAD/PUT/POST/DELETE) over TCP; no fixed URIs — everything discovered via links from DeviceCapability (§4.2). "Client" = whoever interacts with a resource; "server" = whoever exposes it (§4.2) — same device can be both (the hub is a client northbound, and becomes a *server* only if it accepts Subscription Notifications).
- Hub's role mapping: IEEE 2030.5 **client** (DER gateway/ESI-facing device) of a utility server, exercising DeviceCapability, Time, EndDevice/Registration, FSA, DER, Metering Mirror (MUP posting), Flow Reservation, Pricing, Response function sets. The formal "aggregator" role is defined by **CSIP (SunSpec Common Smart Inverter Profile)**, which profiles this standard; 2030.5 itself is HAN-oriented and topology-agnostic (see "aggregator/gateway" notes below).

## Security requirements (clauses 5–7)

**Transport & payloads**
- HTTPS = HTTP over TLS **1.2** (RFC 2818 + RFC 5246); required for all resource access needing auth/confidentiality/integrity (6.4). HTTP-port requests are unauthenticated by definition (AuthType 0x1).
- **Mandatory cipher suite: TLS_ECDHE_ECDSA_WITH_AES_128_CCM_8 (RFC 7251) on curve secp256r1** (6.7). Clients SHALL support it for validating ECC certs and SHOULD support the server's request for an ECC client cert; extra suites MAY be added at equal-or-better strength. (Hub's wolfSSL `ECDHE-ECDSA-AES128-CCM-8 TLSv1.2` matches exactly.)
- Media types: `application/sep+xml` and `application/sep-exi` only (§4.2). Servers SHALL support both encodings; **clients SHALL support XML or EXI (either one)** (5.6.1). XML 1.0, UTF-8, no XML declaration (5.6.2). EXI: schema-informed non-strict grammar, schemaId "S1", bit-packed defaults (5.6.3). `Accept` header mandatory; for sep-exi an extensibility level `level=(-|+)S1` SHALL be specified (5.7.2).
- HTTP headers (Table 1): MANDATORY = Accept, Allow, Content-Type, Date, Host, Location (in POST/201 responses); Content-Length "required in many situations"; ETag/Range/Content-Range optional (file download); nearly everything else DISCOURAGED — including **Transfer-Encoding** (response, discouraged — relevant to the hub's chunked-decode AD-009: compliant servers are discouraged from, but not forbidden, chunking). All response-code handling SHALL be RFC-compliant (5.5.1); minimal client may act on first digit only (5.5.3). Key codes: 200/201(+Location)/204 (success; 204 for empty representation), 301/302 (→ rediscovery; 302 for HTTP→HTTPS redirect), 400, 401 (server may prefer 404), 404 (also used in lieu of 401), 405 (+Allow), 406, 413, 500, 503.
- URIs ≤255 bytes (SHOULD ≪80); clients SHALL NOT assume URI structure — always follow links; MAY cache URIs but on unexpected result SHOULD rediscover (§4.2, 4.5). List paging via `s`/`a`/`l` query params; default limit **1**; lists SHALL support s/l (time-keyed lists also `a`); clients MUST NOT index into list URIs (4.6–4.7).

**Device credentials (6.3)**
- Certificate fingerprint = SHA-256 over the **whole DER-encoded certificate** (6.3.2).
- **SFDI** = fingerprint left-truncated to 36 bits, shown as 11 decimal digits + sum-of-digits check digit (digits incl. check ≡ 0 mod 10) (6.3.3).
- **LFDI** = fingerprint left-truncated to 160 bits (40 hex digits) (6.3.4) — globally unique ID used e.g. in Responses (`endDeviceLFDI`). Note: because LFDI hashes the full cert, ANY reissued cert changes LFDI (matches the hub's CERT_ROTATION_RUNBOOK caveat).
- Optional 6-digit PIN (5 + check digit); SHALL NOT be used to derive keys (6.3.5). Registration code = SFDI‖PIN (6.3.6).

**PKI & certificates (6.11)**
- Manufacturing PKI paths: SERCA>device, SERCA>MICA>device, SERCA>MCA>MICA>device (6.11.3.1). Certificates have **indefinite lifetime**; no CRLs, no OCSP for 2030.5 certs — validity = signature chain to a stored SERCA root ("IEEE 2030.5 Cert - Indef", 6.11.2); OCSP allowed ONLY for non-2030.5 (generic) certs (6.11.10.2).
- Device requirements (6.11.9): private keys non-exportable (SHOULD); SERCA roots embedded at manufacture, not over-the-air replaceable except via firmware upgrade (SHALL); device SHALL present full chain minus SERCA in the TLS handshake; complete valid chain embedded at manufacture.
- Verification per RFC 5280 §6; MUST reject unsupported critical extensions, policy mappings, name constraints (6.11.10.1). RFC 5280 + IEEE 802.1AR (IDevID) compliance (6.11.4).
- Cert content: deviceType OID in CertificatePolicies {1=generic device, 2=mobile (multi-ESI-domain), 3=post-manufacture} (6.11.7.2); policy OIDs {1=test cert, 2=self-signed, 3=service provider, 4=bulk-issued} (6.11.7.3); SubjectAltName HardwareModuleName (RFC 4108) hwType OID + hwSerialNum, combination unique (6.11.7.4). Device cert profile in 6.11.8.3.3; self-signed and generic client/server certs for non-native entities in 6.11.8.4.
- Server hosts SHALL use a device certificate presented in the TLS handshake (6.5). Client auth levels: 0x8 device cert / 0x4 self-signed / 0x2 user / 0x1 none.

**Authorization (6.2, 6.6, 6.8)**
- Per-resource ACL model (implementation-abstract): Method bitmap, AuthType bitmap, DeviceType; all three must pass; failure → 400/405 (method) or 404 (SHOULD, in lieu of 401) — i.e. a client must expect 404 for resources it isn't authorized on.
- Default security policy (Table 12, certification baseline): DeviceCapability = AuthType 0xf, no cert, no registration; SelfDevice/EndDevice = 0xc, registered; **FSA, Sub/Notif, Response, Time, DeviceInfo, Power, Network, LogEvent, Configuration, SW download, DRLC, Metering, Billing, Prepayment, Flow Reservation, DER control = AuthType 0x8 (device cert) + registered device**; Pricing & Messaging = 0xc (device or self-signed cert), registered.
- Registration procedure (6.9): out-of-band SFDI(+PIN) provisioning → EndDevice resource creation → client TLS access binds SpecificID ACL → client verifies PIN → cert reused everywhere.
- Security LogEvents (6.10): SEC_TLS_ALERT (logEventID = TLS alert value), SEC_REGISTRATION_MISS, SEC_ACL_ACCESS_FAILED (all SHOULD).

**Discovery (clause 7)**
- DNS-SD (RFC 6763) with mDNS (.local) and normative **xmDNS** (.site., site-local multicast FF05::FB) for in-premises discovery; unicast DNS-SD SHOULD be used for off-link servers **only when a target is explicitly provided out of band** (7.1) — the hub's provisioned-utility-server model is the sanctioned OOB path (options: FSA redirect, OOB DNS name, or OOB name/IP + DeviceCapability URI).
- Service name `_smartenergy._tcp`; instance names ≤63 bytes ending in SFDI (12 digits incl. check, no hyphens). TXT record keys (Table 16): `txtvers=1` (first key; else discard record), `dcap` (mandatory, relative URI to DeviceCapability), `path` (subtype responses), `https={port}` (present ⇒ use HTTPS; absent ⇒ HTTP; empty ⇒ 443), `level` (±S1). Subtype PTR per advertised function set: bill, derp, dr, edev, file, msg, mup, ppy, rsps, sdev, tm, tp, upt; per-remote-device `edev-{SFDI}` subtype (7.5).
- Discovery procedure (7.6): DNS-SD → TLS if required → GET DeviceCapability → follow links. Clients SHALL dynamically discover URIs; SHALL rediscover on DNS-SD change or 404; SHALL follow redirects per RFC 7231.

## Event/Response semantics

**Event model** (Event base type: mRID, description, creationTime, interval{start,duration}, EventStatus, replyTo, responseRequired; RandomizableEvent adds randomizeStart/randomizeDuration). Event-bearing sets: DRLC (EndDeviceControl), DER (DERControl), Pricing (TimeTariffInterval), Messaging (TextMessage), Flow Reservation (FlowReservationResponse).

**EventStatus.currentStatus** (Annex B): 0 = Scheduled; 1 = Active (set at earliest Effective Start Time); 2 = Cancelled (cancel immediately); 3 = Cancelled with Randomization (cancel using max(|randomizeStart|,|randomizeDuration|) as end randomization; only for RandomizableEvent); 4 = Superseded (terminate immediately, start the new event; obey new event's start randomization if within old start-randomization window). Servers keep cancelled/superseded events available for the Effective Scheduled Period (SHOULD).

**Client event rules (10.2.3.3, normative highlights)**
- Poll event lists ≥1×/15 min (SHOULD 5 min) if not subscribed (a); monitor active events for status changes at same cadence (b).
- Events are immutable except status; servers cancel/supersede rather than edit (c).
- One event at a time per function set with direct control (DRLC, DER, Flow-Res same subject mRID, Pricing) (d); superseding event completes the current one (e); tie-break Nested/Overlapping events from equal-primacy servers by larger creationTime (f).
- Completed nested event does NOT reinstate the containing event (i).
- Expired event (Specified End Time < now): ignore; if responseRequired, POST status 254 "received after expired" (l).
- Late-start event (Effective Start < now < Specified End): start now + |randomization|, preserve Specified End Time, apply duration randomization to abbreviated duration (m).
- Overlap handling (n): scheduled-previous → POST "superseded" (7) then ignore; active-previous → switch state at overlapping event's effective start and POST 7.
- Successive events: earlier event's Effective End = later event's Effective Start; randomization SHALL NOT create gaps or conflicts (o); gaps allowed only for non-successive events (p).
- deviceCategory-subset supersession — per-category, with potentiallySuperseded(Time) flags (q, k: on potentiallySupersededTime change + "Partially Superseded" status, client SHALL re-check all possibly-superseding events).
- **DERControl-specific (t)**: distinct controls (op-modes) are independent — overlap/nest without superseding; supersession is per-control; controls not re-listed in the newer event continue from the older event; DERControls always overlap DefaultDERControl.
- Event removed from server ≠ cancelled — act only on explicit status (s).

**Randomization (10.2.4)**: clients acting on events SHALL support it; apply when attributes present; signed bound, uniform pick in [0, bound]; negative = before start / shorter duration. Fixed pseudorandom values allowed but SHALL scale as a percentage of the bound and be distributed across a device population. Cancellation of an active event → apply max(|rS|, |rD|); local user override executes without randomization. Pricing clients need not honor the sign but SHALL honor the magnitude (10.5.3.4).

**Response function set (8.8, Table 27)** — status codes (applicability: D = DRLC+DER, P = Pricing, M = Messaging):
1 event received (bit0; D,P,M) · 2 event started (bit1; D,P,M) · 3 event completed (D,P,M) · 4 opt-out (D) · 5 opt-in (D) · 6 event cancelled (D,P,M) · 7 event superseded (D,P,M) · 8 partial completion, user opt-out (D) · 9 partial completion, user opt-in (D) · 10 completed, no participation (D) · 11 user acknowledged (bit2; D,P,M) · 12 cannot be displayed (M) · 13 aborted — alternate **server** event (D,P,M) · 14 aborted — alternate **program** event (D,P,M) · 252 rejected — control parameter not applicable (D) · 253 rejected — invalid event/out-of-range control (D) · 254 rejected — received after expired (D,P,M). 15–251, 255 reserved. Response contains endDeviceLFDI, status, subject mRID; POST to event's replyTo; server hosting events need not host the response server. (The hub's "CannotComply" flow maps onto 252/253 at receipt and opt-out/abort codes during execution.)

**Multi-server & primacy (10.2.5)**: registered devices SHALL NOT act on *public* DRLC or DER servers (10.2.5.6). Primacy precedence 0>1>2>3 (in-home > contracted SP > non-contracted...); equal primacy → normal event rules. Duplicate events across servers detected by mRID; respond to one. Cross-server/program aborts → Response 13/14. Each event server SHALL also serve Time; clients executing events use the same-FSA Time resource (10.2.5.3, 9.2.3).

## Subscription vs polling

- **Polling is the default and universal fallback** (§4.2); function-set top resources carry `pollRate` (default 900 s). Sleepy/intermittent clients SHALL default to pull.
- Function-set-specific polling floors/ceilings: event lists ≥1×/15 min (10.2.3.3 a/b); FSA ≥1×/24 h, ≤1×/h (8.6.3); Pricing polled clients ≤1×/h, ≥1×/24 h + pre-activation re-check (10.5.3.6); Time ≤1×/15 min post-sync (9.2.3); EndDevice touch ≥1×/48 h recommended (8.5.3).
- **Subscription is optional** per-resource (`subscribable` attribute). Key client duties if used (8.7.3.4): renew ≤24 h/≥1 h; check subscriptions after connectivity loss AND poll subscribed resources after recovery (f, g); URI change terminates subscription (h); Notification payload = full GET representation (8.7.3.2); return HTTP 400 to unwanted Notifications (o); server rate-caps Notifications to 1/30 s/resource/client (k — so subscription is NOT a low-latency guarantee). Subscription management SHOULD use TLS (q). A subscribing client must run an HTTP(S) **server** endpoint for Notifications — a security-surface tradeoff for a product like the hub (polling avoids any northbound listener).

## Aggregator / gateway topology notes

- IEEE 2030.5-2018 has **no formal "aggregator" role**; the nearest constructs: (a) ESI (energy services interface) as the registration/EndDevice host concept (6.9.1); (b) per-remote-device `edev-{SFDI}` DNS-SD records for servers fronting multiple devices (7.5); (c) Metering Mirror explicitly designed for one device posting data on behalf of another (10.11); (d) DERAvailability "MAY be exposed by devices that report on behalf of other devices" (10.10.4.4.5); (e) opModTargetW/opModTargetVar noted as "likely more useful for aggregators" (Annex B); (f) deviceType OID 2 "mobile" for entities homed to multiple ESI domains (6.11.7.2). A gateway like the hub represents each DER (or itself as a combined DER, 10.10.4.4.1 allows either) via EndDevice instances + DER sub-resources on the utility server; the client-side aggregation contract (one EndDevice per DER vs one per gateway, in-band cert vs LFDI assignment) is specified by **CSIP**, not this document — audit CSIP separately for those obligations.
- Multi-server coordination (10.2.5) is the standard's only "topology" machinery: primacy, public-vs-registered servers, duplicate mRID detection, per-FSA time binding.

## Not applicable to the hub's roles (and why)

- **Prepayment (10.8)** — credit-conditioned service delivery for meters/IPDs; no utility program relevance to a DER gateway.
- **Messaging (10.6)** — human-display text events; hub has no display (would only matter if a program mandates message acknowledgment relay).
- **Billing (10.7)** — informational display/true-up data; hub's economics come from Pricing, not Billing.
- **Power Status / Network Status (9.4/9.5)** — optional self-diagnostics; hub reports health via its own /status-API instead; only needed if a utility requires them under EndDevice.
- **File Download (9.8)** — hub firmware is managed via its own deploy pipeline; only relevant if utility-managed OTA were contracted.
- **DRLC (10.3)** — thermostat/appliance load control; DER management (incl. EVSE-as-storage-DER and Flow Reservation) is the hub's control path. Becomes conditional if the hub ever enrolls as an EndDeviceControl target.
- **Self-signed / generic client certificates (6.11.8.4.3-4)** — hub uses Manufacturing-PKI device certs; self-signed paths are for non-native entities.
- **Server-side ACL implementation (6.2), Response storage (8.8.3.2), server list-serving rules** — the hub is the client on the northbound interface; these bind the utility server. Exception: if the hub subscribes (8.7), it becomes a Notification *server* and inherits TLS-server + DeviceInformation-serving obligations (9.3.3 exempts Notification-only servers from DeviceInformation).
- **xmDNS HAN discovery (7.1)** — the hub's utility server is provisioned out of band (the standard's sanctioned alternative, 7.1); xmDNS would only matter for in-premises 2030.5 service advertisement, which the hub does not do. (Conformance test programs may still exercise DNS-SD client behavior — verify against the target certification profile.)
- **mDNS/HAN sleepy-device provisions, battery-powered client rules** (§4.2, 7.1) — hub is mains-powered and always-on.
- **Proprietary extension authoring (clause 11)** — hub emits only standard-namespace payloads; rules apply only defensively (ignore foreign-namespace extensions in server responses, R31).
