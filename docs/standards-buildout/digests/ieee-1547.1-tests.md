# IEEE 1547.1-2020 — Interop/comms-relevant digest for LEXA hub coverage audit

## Document identity

- **IEEE Std 1547.1-2020** — "IEEE Standard Conformance Test Procedures for Equipment
  Interconnecting Distributed Energy Resources with Electric Power Systems and Associated
  Interfaces." Companion conformance-test standard to IEEE 1547-2018.
- Source text: `scratchpad/standards/txt/15471-2020.txt` (16,700 lines). Interoperability tests =
  **Clause 6** (lines ~7909–9077). Note: the task brief said "6.10.5" — there is no such clause;
  the comm-interface test material is Clause 6 (6.1–6.8), plus Commissioning Test **CT5**
  (8.3.3.5, 8.5.3.5) and plant-controller provisions sprinkled through Clauses 5 and 8.
- Structure: Cl.4 general/test-equipment; **Cl.5 type tests (electrical)**; **Cl.6 interoperability
  tests**; Cl.7 production tests; Cl.8 evaluations + commissioning; Cl.9 periodic tests;
  Annex E (informative IEC 61850-7-420 usage), Annex F (HIL, incl. F.2 controller-HIL for
  supplemental DER devices/controllers).
- LEXA framing: 1547.1's EUT is the **DER** (or a plant controller fronting it). The LEXA hub maps
  to the defined "**plant controller** … may function as the DER's interoperability interface"
  (definitions, lines 996–1000). Its northbound 2030.5 client is exactly the DER-side endpoint
  6.8.2 exercises; its southbound SunSpec master plays the same role as the test's "SunSpec
  compliant Modbus master" in 6.8.1.

## Interoperability test procedures (Clause 6)

Protocol options (6.1, Table 41): DER must support **at least one** of IEEE 2030.5-2018,
DNP3 (IEEE 1815-2012 + DNP3-AN-2018-001), or SunSpec Modbus; every supported protocol is tested
against every requirement. The local DER comm interface "shall be available … at all times that
the DER is operating." Product-family results carry across 50–200% power ratings.

Approach (6.2): read-only info → value verified correct; read-write info → **write, read back,
verify equal, then verify functional behavior for ≥2 different values per adjustable setting**.
Functional criteria are borrowed from the Clause 5 functional tests (no new criteria).
Explicitly **excludes malformed/invalid-value messages** (LEXA's mayhem QA goes beyond 1547.1).

| Clause | What's exercised | Pass criteria |
|---|---|---|
| 6.3.1 general | Every request/response in 6.4–6.8 | DER replies within the communication response time of **Table 42 in IEEE 1547-2018** (the latency criterion — no separate timing table in 1547.1) |
| 6.3.2/6.3.3 adjustable-settings procedure | (a) disable function via interface; (b) write settings → read back → verify match + enabled → verify functional effect per the referenced Cl.5 criteria; ≥2 values per setting; (c) disable and verify disabled | Enable/disable works; all settings readable; readback == written; functional tests under written settings meet Cl.5 criteria |
| 6.4 Nameplate data | Read every item of Table 28 in 1547-2018 (ratings, categories, mfr/model/serial/version) | All values match manufacturer-provided expected values; any mismatch = fail |
| 6.5 Configuration info | Read/write Table 42 (of 1547.1) params: active/apparent power rating, var inject/absorb max, charge ratings (storage only) — each **set to 80% of initial**, then restored | Read values match; **DER measured behavior matches the adjusted setting** (verified by independent test instrumentation, not the DER's own interface) |
| 6.6 Monitoring info | Read W, var (inj/abs), V, Hz, Operational State, Connection Status, Alarm Status at two operating points (e.g., W at 20–30% then 90–100% of rating; V ≤0.90 pu then ≥1.08 pu; f ≤57.2 Hz then ≥61.6 Hz); wait ≥30 s before reading | Reported values match actual operating conditions within **Table 3 of 1547-2018 steady-state accuracy**; state/status/alarm flags match true device condition; a reading older than the required response time must return an **error**, not stale data |
| 6.7 Management info | The 11 control-mode tests of Table 44 (see next section), each run per 6.3.2 against the settings tables 10.6.2–10.6.12 / Tables 30–40 of 1547-2018 | Per 6.3.3 + the referenced functional-test criteria clause |
| 6.8 Protocol mappings | Per-protocol data-point mapping for all the above | No vendor-specific/extra data points may be required; comm-interface blocking/locking (proprietary lockout) is **not allowed** |

Protocol-specific setup (6.8):
- **6.8.1 SunSpec Modbus**: test uses a SunSpec-compliant **Modbus master** against the DER
  (RTU or TCP); Ethernet or RS-485 physical layers; **TLS shall not be used**. Data points are the
  IEEE-1547 SunSpec DER models: **model 1** (Mn/Md/SN/Vr), **701** monitoring (W, VAR, PPV/PhV, Hz,
  St, Alrm), **702** nameplate/capability (WMaxRtg, VAMaxRtg, VarMaxInj/AbsRtg, WChaRteMaxRtg,
  VNom/VMax/VMinRtg, NorOpCat/AbnOpCat…), **703** enter-service (PmtSvc, ESVHi/Lo, ESHzHi/Lo,
  ESDlyTms, ESRndTms, ESRmpTms), **704** PF/fixed-var/W-limit (PFWInjEna, PFWInj.PF/.Ext,
  FixedVarEna, FixedVar, MaxLimWEna, **MaxLimW**), **705** volt-var (Ena, VRef, VRefAuto, VRefTms,
  VoltVar.Crv.Pt, RspTms), **706** volt-watt, **707/708** LV/HV trip curves, **709/710** LF/HF trip
  curves, **711** freq droop (DbOf/DbUf/KOf/KUf/RspTms), **712** watt-var.
  *(Audit note: this is the 700-series DER model set — distinct from the legacy 100-series models a
  SunSpec poller may target; relevant to what lexa-modbus/lexa-proto `sunspec` covers.)*
- **6.8.2 IEEE 2030.5**: an IEEE 2030.5-2018-compliant **server** interfaces to the DER under test
  — i.e., the DER side (LEXA's role) is the **client**. TCP/IP + Ethernet required; server "shall
  utilize the **default security policy** defined in IEEE 2030.5-2018" (TLS w/ the CSIP cipher —
  matches LEXA's ECDHE-ECDSA-AES128-CCM-8 invariant). Nameplate = `DERCapability::rtg*` +
  `DeviceInformation::mf*`; monitoring = MUP `ReadingType` codes (accumulationBehaviour=12,
  commodity=1; uom 38=W, 63=var, 29=V, 33=Hz; flowDirection 19/1) + `DERStatus::
  operationalModeStatus/genConnectStatus/storConnectStatus/alarmStatus/stateOfChargeStatus`;
  controls = `DERControl::opMod*` (mode-enable = an **Active Event**), settings =
  `DERSettings`/`DefaultDERControl::setES*`, curves = `DERCurve`/`CurveData`.
- **6.8.3 DNP3**: outstation per DNP3-AN-2018-001 profile; TCP/IP + Ethernet; **no TLS**; LAN time
  sync; multiplexed generic curves (≥10 curves × ≥4 points, "ALREADY EXECUTING" rejection of
  in-use-curve writes); read-by-class monitoring. Not implemented by LEXA — out of scope unless a
  DNP3 northbound is ever added.

## Comms-commanded control tests (6.7 Table 44 — what's written / what's verified)

All follow 6.3.2: write via the interface, read back, then confirm the **measured electrical
response** meets the referenced functional-test criteria. 2030.5 column = what LEXA northbound
would have to consume/apply as the DER-side client.

| # | Function | Written (2030.5 / SunSpec) | Response verified (criteria clause) |
|---|---|---|---|
| 1 | Constant power factor | `opModFixedPFInjectW` (+ excitation) / 704.PFWInj.PF+Ext, PFWInjEna | Reactive power tracks commanded PF at ≥2 PF values (5.14.3.3) |
| 2 | Volt-var (V/Q curve) | `opModVoltVar::CurveData` + `openLoopTms`, `DERCurve::vRef/autonomousVRefEnable/autonomousVRefTimeConstant` / 705.* | Q-vs-V follows written curve incl. open-loop response time; autonomous-VRef on and off cases (5.14.4.3, 5.14.5.3) |
| 3 | Watt-var (P/Q curve) | `opModWattVar::CurveData` / 712.WattVar.Crv.Pt | Q follows P per written curve (5.14.7.3) |
| 4 | Constant reactive power | `opModFixedVar` / `opModTargetVar` / 704.FixedVar+Ena | Q equals commanded setpoint (5.14.8.3) |
| 5 | Volt-watt (V/P curve) | `opModVoltWatt::CurveData` + `openLoopTms` / 706.* | P curtails per written curve (5.14.9.3) |
| 6 | Voltage trip settings | `opModHVRTMustTrip`/`opModLVRTMustTrip::CurveData` / 707/708 curves | Trip at written V/t points (5.4.2.4, 5.4.3.4) |
| 8 | Frequency trip settings | `opModHFRTMustTrip`/`opModLFRTMustTrip::CurveData` / 709/710 curves | Trip at written f/t points (5.5.1.4, 5.5.2.4) |
| 9 | Frequency droop (freq-watt) | `opModFreqDroop::dBOF/dBUF/kOF/kUF/openLoopTms` / 711.* | P modulates per droop equation above/below nominal (5.15.2.3, 5.15.3.3) |
| 10 | Enter service / cease-to-energize (**permit service**) | `opModEnergize`; `DERSettings`/`DefaultDERControl::setESHighVolt/LowVolt/HighFreq/LowFreq/Delay/RandomDelay/RampTms` / 703.* | DER enters service only inside written V/f window after delay/ramp; disabling permit service ceases energization; disabled state persists power cycle (5.6.4) |
| 11 | Limit maximum active power | **`opModMaxLimW`** / 704.MaxLimW+MaxLimWEna | Measured P ≤ written % of rating; response time per 4.6.2 of 1547-2018 (5.13) |

(There is no test #7 in Table 44. Momentary cessation has a DNP3 mapping (6.8.3.12) only.)

**Direct LEXA overlap**: #11 `opModMaxLimW` is the control LEXA northbound already consumes and
lexa-modbus applies (solar ceiling reconciler); #10 `opModEnergize` = connect/disconnect
(desired-doc Conn); #1/#4 fixed PF / fixed var and #2/#3/#5 curve-based modes are the classic
gap-check axes for both the 2030.5 client (which opMod* it parses from a DERControl) and the
SunSpec layer (whether 704/705/706/711/712 models are readable/writable).

## Gateway-relevant setup requirements (plant controller / managing entity)

- **Definition** (lines 996–1000): *plant controller* = control system that manages and commands
  the DER units and supplemental devices, collects their measurements/status, "**may function as
  the DER's interoperability interface**," and may itself perform any IEEE 1547 control function.
  A plant controller is a *supplemental DER device*; DER system/composite certification includes it.
- **Non-power-producing EUT provision** (recurs in 5.13, 5.14.x, 5.15.x, e.g. lines 5736, 6822):
  "Where the EUT is DER equipment that does not produce power, such as a plant controller, the
  EUT's **commanded active power / commanded PF / commanded reactive power** may be used to verify
  operation," with signal injection permitted and a design evaluation/commissioning test to verify
  the installed DER. I.e., a gateway EUT is judged on the correctness of the setpoints it emits —
  exactly the LEXA hub's testable surface (desired-doc contents vs the 2030.5 control in force).
- **5.19 Settings persistence**: 3×3 randomly chosen adjustable parameters (Tables 30–40 of
  1547-2018) written, power-cycled ≥60 s, read back — must match. "May be run using **only the DER
  control component responsible for storing parameter settings**" (i.e., the controller alone).
  LEXA analog: retained MQTT desired docs / config persistence across service restarts.
- **6.1/6.8 availability & openness**: comm interface available whenever the DER operates; no
  proprietary interface blocking/locking; no extra vendor data points may be needed to interoperate.
- **Commissioning CT5 — Interoperability** (8.3.3.5 PoC-composite; 8.5.3.5 PCC-composite): verify
  SW/FW revisions meet 1547; verify the **area EPS operator can access and read and/or write** the
  Clause-10 parameters through the deployed comm channel; 8.5.3.5 adds an explicit field test —
  read nameplate over the comm interface and compare to documentation (pass iff identical).
- **Installation evaluations** (8.3.2.1/8.4.2.1, lines 11164–11172, 12504–12513): confirm
  end-to-end wiring/continuity **from the DER plant controller to the measurement devices and DER
  equipment**; verify external inputs (permit service, active power limit, operating modes) applied
  at the input point reach the DER with performance/response times per **4.6 of IEEE 1547-2018**;
  commissioning criteria include the plant controller measuring V/I/f/P/Q within accuracy and
  isolation devices disconnecting **when commanded by the plant controller** (8.5.3.1.4).
- **Design evaluation split by type-test status** (lines 13700–13712): if a 1547 function is
  performed by a type-tested controller + type-tested DER, evaluate settings aggregation and
  **coordination of communication/IO between plant controller and DER**; a non-type-tested
  controller performing the function requires further evaluation (controls review/bench test); a
  non-type-tested DER doing so requires the full type test on the composite.
- **Annex F.2 (informative)**: controller-HIL — supplemental DER devices (controllers) tested
  against a real-time simulator through their low-power I/O; the pattern for bench-testing a
  gateway without power hardware (csip-tls-test's sim bench is this in spirit).

## Out of scope for the hub (pure electrical type tests)

Electrical performance of the power-conversion equipment — a comms gateway is never the EUT here
except via the plant-controller provisions above. Only the **settings-write/read surface** of these
functions (Clause 6 + Table 44 mappings) touches the hub.

- 5.2 Priority of responses; 5.3 Temperature stability.
- 5.4 / 5.5 Voltage & frequency disturbance response — trip accuracy, ride-through (LVRT/HVRT,
  LFRT/HFRT, ROCOF, phase-angle jump), momentary cessation *(settings write = in scope; measured
  trip/ride-through behavior = out)*.
- 5.6 Enter service and 5.7 Synchronization electrical performance (paralleling, sync accuracy)
  *(permit-service/ES-window settings write = in scope)*.
- 5.8 Interconnection integrity (dielectric, EMI, surge withstand).
- 5.9 DC injection limits; 5.10 **Unintentional islanding** (+ Annex D, G); 5.11 Open phase;
  5.12 Current distortion / harmonics (RSS current tests).
- 5.13–5.15 electrical response measurement (power modulation accuracy, response times of the
  DER itself) *(the adjustable-settings interfaces of these = the in-scope interop payloads)*.
- 5.16 Prioritization of DER responses; 5.17 Limitation of overvoltage contribution;
  5.18 Fault current characterization.
- Clause 7 Production tests (per-unit factory V/f trip checks).
- Clause 8 commissioning other than CT5/IE-comms items: visual/electrical CT1–CT4, CT6 islanding,
  RVC/flicker/power-quality studies.
- Clause 9 Periodic interconnection tests (field re-verification of trip settings/protection).
- Annexes A (test signals), C (measurement uncertainty), G (short-circuit simulator), H (alternate
  ac sources), I (open/closed-loop response-time math) — test-lab apparatus, not gateway behavior.

## Audit hooks (LEXA-specific takeaways)

1. The hub's northbound already implements the CLIENT side of the 6.8.2 surface for a subset:
   `opModMaxLimW`, `opModEnergize`(connect/disconnect), fixed-var/PF and curve modes are the axes
   to gap-check in `lexa-proto/csipmodel` + `internal/northbound` parsing.
2. Southbound coverage question: does the `sunspec` package know the **700-series** models
   (701–712) or only legacy models? 1547.1 conformance of any fronted DER speaks 700-series.
3. Timing: every interface interaction must meet 1547-2018 Table 42 response times; monitoring
   responses must be fresh-or-error (no silent stale data) — cf. LEXA staleness gates.
4. Settings persistence (5.19) and permit-service-disable persistence (5.6) map to retained-topic
   re-seeding — a demonstrable equivalence worth documenting for any composite certification.
5. 1547.1 explicitly does NOT test malformed/invalid messages (6.2) — LEXA's NaN-reject/decode
   alarms and mayhem suite exceed the standard; note as "beyond-conformance" in the audit.
