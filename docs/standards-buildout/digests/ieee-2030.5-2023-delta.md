# IEEE 2030.5-2023 — 2018→2023 revision delta digest

Audience: LEXA hub standards-coverage audit. Hub role here = 2030.5/CSIP **client**
(DER gateway to a utility server, wolfSSL mTLS, XML encoding). Source: full text of
IEEE Std 2030.5-2023 as amended, 398 pp.

## Document identity

- **IEEE Std 2030.5-2023**, "IEEE Standard for Smart Energy Profile Application Protocol".
  Revision of IEEE Std 2030.5-2018; approved 6 Dec 2023. The text INCORPORATES
  **IEEE Std 2030.5-2023/Cor1-2024** (corrigendum approved 26 Sep 2024) — all Cor1 fixes
  are already merged into this text (Foreword, ~p.11).
- Drivers named in the Foreword: **California Rule 21 / Smart Inverter Working Group and the
  IEEE 1547 revision**; the standard states it "supports all of the needs of IEEE Std 1547-2018"
  (Overview, §1). New informative **Annex E** maps 2030.5-2023 parameters to 1547-2018
  (nameplate → DERCapability::rtg*, monitoring → ReadingType tuples, etc.).
- **Schema version 2.2**: every XML payload SHALL carry `schemaVer="2.2"` on the top-level
  element (§5.6.2); absence ⇒ assumed 2.1 (i.e., a 2018 device). EXI schemaId is **"S2"** (§5.6.3).
  **XML namespace UNCHANGED**: `urn:ieee:std:2030.5:ns` — deliberately, per the §4.9
  compatibility design (a namespace change is reserved for a future compatibility break, §4.9).
- §4.9 formalizes forward/backward compatibility rules: clients ignore unknown elements/values;
  new revisions only add optional elements. One acknowledged exception: the
  mandatory→optional DERControl-mode change "is not strictly backward compatible" (§10.10.4.2.1).

## Function-set catalog (2023) — confirmation

Support (cl. 8): DeviceCapability · SelfDevice · EndDevice · **ProxiedDevice (NEW)** ·
**AggregatedDevice (NEW)** · FunctionSetAssignments · Subscription/Notification · Response.
Common (cl. 9): Time · DeviceInformation · PowerStatus · NetworkStatus · LogEvent ·
Configuration · FileDownload.
Smart energy (cl. 10): DRLC · Metering · Pricing · Messaging · Billing · Prepayment ·
FlowReservation · **DER (heavily extended)** · Metering Mirror.
Identical to 2018 except the two NEW clause-8 function sets and the DER extensions below.

## 2018→2023 delta inventory

Numbered; **[client: yes/no]** = does it impose a new requirement on a client device.

### DER function set (§10.10, Annex B.23) — the bulk of the revision

1. **No DERControl modes are mandatory anymore** (§10.10.4.2.1). 2018 designated some modes
   (opModEnergize etc.) mandatory; 2023 makes ALL modes optional "to support a variety of
   regional and interconnection requirements", and notes this is *not strictly backward
   compatible*. **[client: no — a relaxation; interconnection rules (CSIP, Rule 21) re-impose sets]**
2. **New DERControl modes, DERControlType bits 27–31** (marked "added in IEEE Std 2030.5-2023"):
   `opModDeltaVar`, `opModDeltaW` (relative-change controls, with new
   `DeltaBidirectionalType` governing charge/discharge sign crossing), `opModFixedV`,
   `opModGridConnectPermit`, `opModIslandPermit` (microgrid connect/island permission).
   **[client: only if supported — advertise via modesSupported]**
3. **New bitmap `DERControlType2`** (entirely new, 10 modes): `opModMaxLimPctVAAbsorb/Inject`,
   `opModMaxLimPctVarAbsorb/Inject`, `opModMaxLimPctWAbsorb`, `opModMaxLimVarAbsorb/Inject`,
   **`opModMaxLimWAbsorb` / `opModMaxLimWInject` (absolute-watt import/export limits — 2018 only
   had the percent-based opModMaxLimW)**, `opModTargetV`. Carried in new
   `DERCapability::modesSupported2` / `DERSettings::modesEnabled2` /
   `DERControlResponse::modesResponded2`. **[client: only if supported]** Note kept in text:
   opModMaxLimW stays PerCent "for historical reasons"; its preferred name would have been
   opModMaxLimPctWInject.
4. **Ride-through / ES semantics tightened for 1547-2018**: must-trip / may-trip / momentary-
   cessation region definitions (§10.10.4.3.2) with worked LFRT/HFRT curve tables;
   `DefaultDERControl` setES\* (enter-service) values now **SHALL update the corresponding
   DERSettings value** when present (Annex B DefaultDERControl), i.e., server-pushed
   enter-service reconfiguration. DefaultDERControl also gains `updatedTime`. **[client: yes if
   DER-managing — must apply pushed ES settings into its settings state]**
5. **DERCurve extended** (Annex B.38): new `autonomousVRefEnable` + `autonomousVRefTimeConstant`
   (volt-var autonomous vRef adjustment — 1547 VV mode; a DER unable to do autonomous vRef
   SHALL still execute the curve without it), `openLoopTms`, `rampPT1Tms`, `vRef`, and
   **`yRefType` (DERUnitRefType)** replacing fixed y-axis meaning. `DERUnitRefType` gains values
   4–8: %setEffectiveV, %setMaxChargeRateW, %setMaxDischargeRateW, %statWAvail, %setMaxVA —
   storage-aware percent bases (negative %setMaxW values now read against setMaxChargeRateW).
   **[client: yes if executing curve modes — must interpret yRefType/new bases]**
6. **Single-DER rule**: `DERList` SHALL contain at most ONE DER object ("previous revisions...
   allowed more than one"); that DER represents the whole EndDevice and is what DERControls act
   on. Sub-parts go in the **NEW `DERComponentList` / `DERComponent`** resources (each component
   has its own lFDI, may have its own (Mirror)UsagePoint) (§10.10.4.4.1, B.23). A combined
   PV+storage system is one DER (DERType 83) with components. **[client: yes — a client that
   POSTed multiple DER entries per EndDevice must restructure]**
7. **NEW `CurrentDERControls` resource** (A.4.10.5, GET/PUT mandatory on the resource): the DER
   reports the mode values it is *currently executing*, regardless of source (DERControl,
   DefaultDERControl, or local), for everything flagged in modesEnabled/modesEnabled2
   (§10.10.4.4.1). Read-only for control purposes — not a command channel. **[client: yes if the
   server hosts it under the client's EndDevice — client PUTs its executing state]**
8. **DERStatus reshaped**: `connectStatus` re-typed to **`ConnectStatusType2`** (bitmap: bit 0
   Connected, bit 1 Energized — matching opModConnect/opModEnergize split);
   `genConnectStatus` and `storConnectStatus` **DEPRECATED — SHALL NOT be included** (2018
   devices may still send them); new `localControlModeStatus` (local vs remote control).
   `stateOfChargeStatus` / `storageModeStatus` (charging/discharging/holding) retained for
   storage. **[client: yes — DERStatus PUTs must use the new shape]**
9. **DERAvailability** gains absorb-side headroom: `statVarAbsorbAvail`, `statWAbsorbAvail`
   (2018 had only the injection-side statVarAvail/statWAvail) — ES/V2G-relevant reserve
   reporting. **[client: optional fields]**
10. **DERControl gains `deviceCategory`** targeting (devices SHOULD ignore events not matching
    their category), aligning DER events with DRLC-style category filtering. **[client: yes —
    must filter]**
11. **Per-mode event overlap semantics codified** (§10.2.3 rule q): DERControl modes within
    overlapping DERControl events are independent; a later event supersedes an earlier one
    *only for the modes (and deviceCategories) it names*; unnamed modes of the original keep
    executing. **[client: yes — event scheduler must supersede per-mode, not per-event]**

### Eventing model (§10.2, Response function set)

12. **EventStatus 4 = "Superseded" DEPRECATED** — servers SHALL NOT use it (clients must still
    tolerate it from 2018 servers); **NEW EventStatus 5 = "Completed"** (server sets after the
    event's max Effective Scheduled Period; "not present in revisions prior to 2023").
    `potentiallySuperseded` (now fixed: SHALL be set true) and `potentiallySupersededTime`
    DEPRECATED. Supersession is now entirely a **client-side** determination from overlap +
    primacy + creationTime rules. **[client: yes]**
13. **Removed-event = cancelled** (§10.2.3 rule p): a client SHALL treat an event that vanishes
    from the server before its Effective Scheduled Period ends as cancelled (and respond
    "cancelled" if responseRequired) — text notes "this behavior differs from previous
    revisions". **[client: yes]**
14. **DefaultDERControl is now respondable** (`RespondableSubscribableIdentifiedObject`); the
    Response-type table (Table 31) extends Received/Started/Cancelled/Superseded/Not-Supported/
    Not-Applicable/Invalid/Expired semantics to DefaultDERControl (e.g., "Received" also fires
    when the client detects a *changed* DefaultDERControl value). New Response subtypes
    **`DERControlResponse`** (`modesResponded` — **SHALL be present** from 2023-compliant
    clients — + `modesResponded2`) and **`DefaultDERControlResponse`** (`defaultsResponded`
    using new `DefaultDERControlType` bitmap over setESDelay…setSoftGradW). **[client: yes —
    response payloads change]**

### Device / support resources

15. **NEW Proxied Device function set** (§8.6): a device (gateway) exposes `ProxiedDeviceList`
    for devices it fronts; sFDI query param; duplicate/removal rules; LFDI derivation for
    non-2030.5 proxied devices defined in §6 (truncated-hash concatenation, traceable per
    device). **[client: optional — but squarely the LEXA gateway topology]**
16. **NEW Aggregated Device function set** (§8.7): `AggregatedDeviceList` under
    SelfDevice/EndDevice; an aggregation acts as a single device; plus NEW
    `AbstractDevice::distribution` (`AggregationDistributionType`: Uniform / Prorate /
    Priority) and NEW `AggregationPriority` resource telling the aggregator HOW to split a
    control across members. **[client: optional; aggregator clients: yes]**
17. **`AbstractDevice::lFDI` now SHALL be included** (optional in 2018) — affects every
    EndDevice/SelfDevice/ProxiedDevice representation a client POSTs. **[client: yes]**
18. **EndDeviceList `sFDI` query-string filter**: servers SHALL process it ("not available in
    revisions prior to 2023"); client usage OPTIONAL. Same for the new ProxiedDevice/
    AggregatedDevice lists. **[client: no — optional convenience]**
19. **`Registration::pollRate` DEPRECATED** (servers SHALL NOT include). EndDevice `postRate`
    (client-preferred POST cadence, server may override) present. **[client: minor — stop
    reading Registration.pollRate]**
20. **Deprecated "Active…List" pattern, wholesale**: `ActiveDERControlListLink`,
    `ActiveEndDeviceControlListLink`, `ActiveFlowReservationListLink`,
    `ActiveTimeTariffIntervalListLink`, `ActiveBillingPeriodListLink`,
    `ActiveCreditRegisterListLink`, `ActiveProjectionReadingListLink`,
    `ActiveSupplyInterruptionOverrideListLink`, `ActiveTargetReadingListLink`,
    `ActiveTextMessageListLink` + the A.x resources behind them, and **`CurrentDERProgramLink`**
    — all DEPRECATED (servers SHALL NOT include; 2018 servers may). Clients compute "active"
    from the full event lists + status/primacy rules. **[client: yes — remove any reliance]**
21. **Time source clarified**: event times are relative to the Time resource of the FSA that
    delivered the event (fallback: host server's Time). **[client: yes — minor, multi-server]**

### Discovery / transport / encoding

22. **mDNS (RFC 6762) is now normative; xmDNS DEPRECATED** (still normative only for backward
    compatibility) — "Beginning with this revision" (§7.1). TXT keys unchanged
    (dcap/path/https/level). **[client: no for LEXA — plain mDNS/DNS-SD already]**
23. **HTTP 429 ("Too Many Requests") added** to the §5.5 response-code catalog; 503+Retry-After
    handling SHOULD be implemented by clients. **[client: SHOULD-level — add backoff on 429/503]**
24. **EXI schemaId "S2"** (2023 schema) vs 2018's grammar; XML/EXI support matrix unchanged
    (servers: both; clients: either). **[client: only if EXI]**
25. `schemaVer` attribute mandatory on all XML payloads (see Document identity). **[client: yes —
    stamp `schemaVer="2.2"` on everything POSTed/PUT once claiming 2023]**

## New client-side requirements (consolidated, for a DER-gateway client)

MUST (to claim 2023 conformance):
- Emit `schemaVer="2.2"` on every XML payload; tolerate/ignore it inbound (§5.6.2).
- Include `lFDI` in every AbstractDevice representation POSTed (EndDevice etc.).
- Present at most ONE DER per EndDevice (components → DERComponentList).
- PUT `DERStatus` in the 2023 shape: `connectStatus` as ConnectStatusType2, no
  genConnectStatus/storConnectStatus.
- Include `modesResponded` in every DERControlResponse; respond to DefaultDERControl per
  responseRequired (incl. on detected value changes).
- Event handling: recognize status 5 (Completed); tolerate-but-don't-require status 4;
  client-side supersession per-mode/per-deviceCategory; treat premature disappearance from the
  server as cancellation; filter DERControls on deviceCategory.
- Stop consuming deprecated Active\*List / CurrentDERProgram links (compute active state
  locally); if the server exposes CurrentDERControls under the client's DER, PUT
  currently-executing modes there.
- Apply DefaultDERControl-pushed setES\*/setGradW/setSoftGradW into DERSettings state.

SHOULD / conditional:
- 429/503 backoff with Retry-After.
- Advertise any of the 15 new modes via modesSupported (bits 27–31) / modesSupported2 and
  execute per the new DERUnitRefType/yRefType percent bases; DeltaBidirectionalType for
  opModDelta\*.
- statWAbsorbAvail/statVarAbsorbAvail in DERAvailability for storage headroom.
- ProxiedDevice / AggregatedDevice function sets if representing fronted/aggregated fleets
  (with AggregationPriority-directed control splitting).

## Security delta

**Essentially none — the 2018 security architecture is retained verbatim:**
- Still **HTTPS = HTTP over TLS 1.2 (RFC 5246)**. No TLS 1.3 anywhere in the document.
- Mandatory cipher suite still exactly **TLS_ECDHE_ECDSA_WITH_AES_128_CCM_8 (RFC 7251) on
  secp256r1** (§6.7); additional suites MAY be offered at equivalent strength.
- Device identity unchanged: LFDI = SHA-256 over the whole DER-encoded cert (§6.3), SFDI with
  checksum digit; Manufacturing PKI with **indefinitely-valid device certs, no OCSP/CRL inside
  the 2030.5 PKI** (OCSP optional for non-2030.5 "generic" certs only, Table 14).
- Six certificate classes enumerated (device, device-test, additional server certs for ESIs,
  generic client/server for non-native entities, self-signed non-native) — taxonomy/matrix
  presentation, not a new client obligation.
- New security-adjacent items are peripheral: LFDI-derivation rule for proxied
  devices/DERComponents (§6.3), `LE_UNAUTHORIZED_WRITE_TO_PROXIED_DEVICE` LogEvent.
- **Consequence for LEXA**: the wolfSSL TLS 1.2 + AES128-CCM-8 stack and LFDI computation carry
  to 2023 unchanged. (CSIP/CA Rule 21 profiles may move independently; that is out of this
  document's scope.)

## Migration assessment for a 2018-conformant client

Overall: **moderate, almost entirely in the DER function set and event state machine; zero in
TLS/certs/discovery-transport.** Same namespace + §4.9 rules mean a well-behaved 2018 client
largely interoperates with a 2023 server today (new elements are optional/ignorable; the sFDI
param is ignored; missing schemaVer reads as 2.1). The work to *claim 2023*:

1. Payload plumbing (cheap): schemaVer stamp; mandatory lFDI; drop deprecated fields from PUTs.
2. DERStatus type change (small but breaking): ConnectStatusType2 bitmap.
3. Response layer (small): modesResponded(2) fields; DefaultDERControl response duty.
4. Event engine (the real work): Completed status, no server-side Superseded, per-mode
   supersession, removal-as-cancellation, deviceCategory filtering — a 2018 event scheduler
   needs a genuine rework, and this is also where 2018-vs-2023 *server* behavior differs most
   (a 2023 client must still tolerate 2018 servers emitting Superseded/potentiallySuperseded).
5. DER model restructure (design-level, if applicable): one DER per EndDevice + DERComponents;
   CurrentDERControls reporting.
6. Optional feature adoption (business-driven): the 15 new control modes — most valuable to a
   storage/EVSE gateway are the absolute-watt **opModMaxLimWInject/WAbsorb** (site
   export/import caps in W, not %) and the microgrid connect/island permits; ProxiedDevice/
   AggregatedDevice as the sanctioned model for a multi-asset gateway.

Risk note: because 2023 makes all DERControl modes optional AND deprecates server-side
supersession, "2023-conformant" says less about actual capability than "2018+CSIP" did — a
utility program spec (CSIP revision) remains the binding capability list.

## Not applicable / no material change (checked)

- Metering, Metering Mirror, Pricing, Billing, Prepayment, Messaging, DRLC, FlowReservation
  function sets: structurally unchanged (only the deprecated Active\*List links above;
  FlowReservation flows and RequestStatus/EnergyRequest model as in 2018 — EV charge
  scheduling unchanged).
- No V2G-specific function set was added; EV support remains DERType 81/82/83 + FlowReservation
  + storage-aware DER modes; Annex D.2 PEV guidance still "subject to work with SAE and ISO/IEC".
- Time, DeviceInformation, PowerStatus, NetworkStatus, LogEvent, Configuration, FileDownload:
  no client-relevant deltas found.
- Subscription/Notification mechanics unchanged (notification servers still need TLS-server
  capability for secure notifications — as in 2018).
- DNS-SD TXT record keys, service names, port defaults, URI/query-string conventions (start/
  after/limit), list-paging: unchanged.
- DERCapability rtg\* / DERSettings set\* fields (incl. setES\*, setGradW, setSoftGradW,
  charge/discharge VA+W ratings, PF over/under-excited pairs): already present in 2018; 2023
  adds only modesSupported2/modesEnabled2 and clarified storage percent-base semantics.
